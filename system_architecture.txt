# U-Share: System Architecture, Data Flow, and ER Descriptions

This document provides the necessary structure, flows, and relationships required to create architectural diagrams, Entity-Relationship (ER) diagrams, and Data Flow Diagrams (DFDs) for the U-Share project.

---

## 1. High-Level Architectural Diagram

**Architecture Type:** Client-Server architecture (MERN Stack)

**Components:**
1. **Client Layer (Frontend)**
   - Framework: React.js (Vite)
   - State Management: Redux Toolkit
   - Styling: Tailwind CSS
   - Deployment: (e.g., Vercel / Netlify)

2. **Network/Transport Layer**
   - REST API (Express.js routes)
   - WebSockets (Socket.io for real-time chat)

3. **Application Layer (Backend)**
   - Runtime: Node.js
   - Framework: Express.js
   - Middleware: JWT Authentication, Multer (File Upload), CORS

4. **Data Layer (Database)**
   - DBMS: MongoDB (Mongoose ODM)
   - Cloud Storage: Cloudinary (Images/ID Cards)

5. **Third-Party Integrations**
   - Payment Gateway: Razorpay
   - Email Service: Resend (OTP/Password Reset)

### Architecture Mermaid Diagram
```mermaid
graph TD;
    Client[Client Browser - React.js] <-->|REST API / HTTPS| Backend[Backend - Node/Express]
    Client <-->|WebSockets| Backend
    Backend <-->|CRUD Operations| DB[(MongoDB)]
    Backend <-->|Upload Images| Cloudinary[Cloudinary API]
    Backend <-->|Process Payments| Razorpay[Razorpay API]
    Backend <-->|Send Emails| Resend[Resend API]
```

## 2. Entity-Relationship (ER) Diagram

### Entities and Attributes

1. **User**
   - `_id` (Primary Key)
   - `name` (String)
   - `email` (String)
   - `password` (String, Hashed)
   - `role` (Enum: 'user', 'admin')
   - `idCardImage` (String, URL)
   - `isVerified` (Boolean)
   - `wishlist` (Array of Item IDs)

2. **Item**
   - `_id` (Primary Key)
   - `owner` (Foreign Key -> User)
   - `title` (String)
   - `description` (String)
   - `pricePerDay` (Number)
   - `category` (String)
   - `condition` (String)
   - `images` (Array of Strings)
   - `isAvailable` (Boolean)

3. **Booking**
   - `_id` (Primary Key)
   - `item` (Foreign Key -> Item)
   - `borrower` (Foreign Key -> User)
   - `owner` (Foreign Key -> User)
   - `startDate` (Date)
   - `endDate` (Date)
   - `totalPrice` (Number)
   - `paymentMethod` (Enum: 'Online', 'COD')
   - `paymentStatus` (Enum: 'Pending', 'Completed')
   - `razorpayOrderId` (String)

4. **Chat**
   - `_id` (Primary Key)
   - `participants` (Array of Foreign Keys -> User)
   - `item` (Foreign Key -> Item)
   - `messages` (Array of Objects: { senderId, text, timestamp })

5. **Review**
   - `_id` (Primary Key)
   - `reviewer` (Foreign Key -> User)
   - `reviewedEntity` (Foreign Key -> User or Item)
   - `rating` (Number 1-5)
   - `comment` (String)

### Relationships
- A **User** can list many **Items** (1:N)
- A **User** can make many **Bookings** (1:N)
- An **Item** can have many **Bookings** (1:N)
- A **User** can participate in many **Chats** (N:M)
- A **User** can write many **Reviews** (1:N)
- An **Item** can have many **Reviews** (1:N)

### ER Mermaid Diagram
```mermaid
erDiagram
    USER ||--o{ ITEM : "lists"
    USER ||--o{ BOOKING : "creates"
    USER ||--o{ REVIEW : "writes"
    ITEM ||--o{ BOOKING : "has"
    ITEM ||--o{ REVIEW : "receives"
    USER }|--|{ CHAT : "participates"
```

## 3. Data Flow Diagrams (DFD)

### Level 0 DFD (Context Diagram)
- **External Entities:** Student (Borrower/Owner), Admin, Payment Gateway (Razorpay).
- **Central Process:** U-Share Marketplace System (Process 0).
- **Flows:**
  - Student -> System: Account details, ID upload, Item details, Search queries, Booking requests, Chat messages, Payment.
  - System -> Student: Available items, Booking confirmations, Messages, Alerts.
  - Admin -> System: Moderation actions, Verification approvals.
  - System -> Admin: Dashboard stats, Pending verifications.
  - System <-> Razorpay: Payment intents, Success/Failure status.

### Level 1 DFD (Main Processes)
1. **User Management (Process 1.0)**
   - Handles Registration, Login, ID Verification. 
   - Reads/Writes to `Users` Database.
2. **Item Management (Process 2.0)**
   - Handles Create, Read, Update, Delete of Listings. 
   - Reads/Writes to `Items` Database.
3. **Transaction/Booking System (Process 3.0)**
   - Handles availability checking, cost calculation, and payment processing.
   - Reads/Writes to `Bookings` Database.
4. **Communication System (Process 4.0)**
   - Handles Real-time messaging via Socket.io.
   - Reads/Writes to `Chats` Database.

## 4. System Flow (User Journey)

**Scenario: Renting an Item**
1. **Start:** User (Borrower) logs in securely (JWT token stored in local/cookie).
2. **Search:** Borrower navigates to `/browse`, applies filters (e.g., Electronics). Frontend calls `GET /api/items`.
3. **Select:** Borrower clicks an item. Frontend calls `GET /api/items/:id`.
4. **Negotiate:** Borrower clicks "Message Owner". Socket connection establishes. Messages are exchanged in real-time and saved to the Chat DB.
5. **Book:** Borrower selects Dates. Frontend calculates total price and calls `POST /api/bookings`.
6. **Payment:** 
   - If 'Online': Backend calls Razorpay API to create order. Razorpay UI pops up on Frontend. User pays. Frontend sends success payload to Backend (`POST /api/payments/verify`).
   - If 'COD': Booking is created directly.
7. **Fulfill:** Owner receives notification. Borrower and Owner meet on campus.
8. **End:** Rental finishes, User leaves a review via `POST /api/reviews`.
